## Decisions made

### 1. xcodegen as the source of truth for the project file

Instead of hand-editing `project.pbxproj` (a 5,000-line generated XML blob that produces noisy git diffs and frequent merge conflicts), the project is **declared in `iOS/project.yml`** and generated via [`xcodegen`](https://github.com/yonaskolb/XcodeGen). Both `project.yml` *and* the generated `LakeloomApp.xcodeproj` are committed:

- Cloning the repo and opening Xcode is one step (no `make project` required for first-time openers).
- Project changes go through PR review on a 70-line YAML diff, not a 500-line pbxproj diff.
- `make project` regenerates from YAML when settings change.

Trade-off: contributors must run `xcodegen generate` (or `make project`) after editing `project.yml`. The Makefile target makes this trivial.

### 2. Bundle ID convention

- Debug → `com.databricks.lakeloom.dev`
- Release-TestFlight + Release-AppStore → `com.databricks.lakeloom`

Uses the `com.databricks.lakeloom` prefix because lakeLoom is an internal Databricks tool. The `.dev` Debug suffix lets a developer install both the local-build and a TestFlight build side-by-side on the same device. Module 10 §6.1 documents this.

If Field Engineering wants a more specific prefix (e.g., `com.databricks.fde.lakeloom`), it's a one-line change in `project.yml` + a re-generate.

### 3. Three build configurations from day one

`Debug`, `Release-TestFlight`, `Release-AppStore` — per Module 10 §5. Differ in:

- `SWIFT_OPTIMIZATION_LEVEL`: `-Onone` (Debug) vs `-O` (both Release variants).
- `SWIFT_ACTIVE_COMPILATION_CONDITIONS`: `DEBUG INTERNAL_TOOLS` (Debug) → `INTERNAL_TOOLS` (TestFlight) → empty (App Store).
- `ENABLE_TESTABILITY`: `YES` for Debug only.
- `PRODUCT_BUNDLE_IDENTIFIER`: `.dev` suffix only on Debug.

Behavior toggles (e.g., enabling the in-app log viewer) are runtime feature flags, not compile-time, so a TestFlight build can flip them without rebuilding (Module 10 §5.4).

### 4. Swift 6 strict concurrency on day one

`SWIFT_VERSION = 6.0` and `SWIFT_STRICT_CONCURRENCY = complete` are baseline in `project.yml`. New code is expected to compile cleanly under it. The actor-based architecture in Modules 01–11 was designed for this; we lock in the constraint now to avoid retrofitting later.

`SWIFT_TREAT_WARNINGS_AS_ERRORS = YES` and `GCC_TREAT_WARNINGS_AS_ERRORS = YES` are also on. Any warning the compiler can emit is one we want to fix immediately, not let drift into background noise.

### 5. iOS 26 minimum, today

Per Module 02, the live STT path uses iOS 26's `SpeechAnalyzer` / `SpeechTranscriber`. Setting `IPHONEOS_DEPLOYMENT_TARGET = 26.0` from the start avoids pretending we support iOS < 26 in any code path. The audience (Databricks employees on FDE-issued iPhones) will run iOS 26+; this is a safe constraint.

### 6. Drop `PRODUCT_NAME = lakeLoom` override

Initial draft of `project.yml` set `PRODUCT_NAME: lakeLoom` so the `.app` bundle would be named `lakeLoom.app`. This caused `xcodebuild test` to fail because xcodegen infers `TEST_HOST` from the target name (`LakeloomApp`), not `PRODUCT_NAME`. Dropped the override; the `.app` bundle is now `LakeloomApp.app`. The user-facing display name is still **lakeLoom** via `INFOPLIST_KEY_CFBundleDisplayName`. The product-name change is invisible to users — only developers see `LakeloomApp.app` in DerivedData paths.

### 7. Zero SPM dependencies in this PR

The architecture overview lists `swift-async-algorithms`, `swift-collections`, and `libopus` as v1 dependencies. None of them are needed yet — the scaffold only renders a static splash. SPM packages will land **alongside the modules that need them**:

- `swift-async-algorithms` arrives with Module 03 (IngestService, for the merged event stream pattern in the drainer).
- `swift-collections` arrives with Module 09 (Telemetry, for the bounded `Deque` log ring buffer).
- `libopus` arrives with Module 02 (CaptureEngine recorder).

Each dependency review is per-module, not pre-loaded. Smaller scaffold PR, more focused future PRs.

### 8. Smoke test + launch test as the v1 testing baseline

- `AppTests/SmokeTests.swift` — Swift Testing-based; asserts the `LakeloomApp` module is `@testable`-importable. Compile success is the assertion shape.
- `AppUITests/LaunchTests.swift` — XCUITest; launches the app and waits for SplashView's combined accessibility label.

Both green on iPhone 17 Simulator with iOS 26.2. Real per-module suites land alongside their modules; these are the floor everything else builds up from.
